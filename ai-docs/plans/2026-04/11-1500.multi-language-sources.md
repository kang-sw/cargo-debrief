# Multi-Language Sources — Phase 1: Glue Layer Rewrite

Scope: Phase 1 of epic `260409-epic-multi-language-sources`. Replaces
the Rust-hardcoded indexing pipeline with a `[[sources]]`-driven pipeline.
Phase 2 (CppChunker body) and Phase 3 (dep source indexing) are out of
scope — their skeletons stay as `todo!()` stubs.

## Context

**What the skeleton already provides (locked, do not redesign):**

- `config::SourceEntry`, `config::Language`, three-layer `Config` with
  `sources: Vec<SourceEntry>`, `append_source`/`remove_source_at` (stubs),
  and `resolve_sources(project_root)` with Cargo.toml backward-compat
  fallback — contract from directive 4.
- `service::DebriefService` trait with 8 methods. All five legacy methods
  (`index`, `search`, `overview`, `dep_overview`, `set_embedding_model`)
  plus three new registration methods (`add_source`, `list_sources`,
  `remove_source`). Trait signatures are frozen — do not touch.
- `chunker::chunker_for(&Language) -> Box<dyn Chunker + Send + Sync>`
  routing. `CppChunker` stub panics via `todo!()` in Phase 1; Phase 1
  must not instantiate it against real `.cpp` inputs.
- `main.rs` CLI parsing is wired: `RebuildIndex { no_deps }`, `Add`,
  `Sources`, `Remove`, `SetEmbeddingModel`, `Search`, `Overview`,
  `Daemon`, plus `__daemon` entry. `init_tracing()` already installs
  `tracing_subscriber::fmt` honoring `RUST_LOG`.
- `store.rs` currently has `INDEX_VERSION = 7`, `DepsIndexData`,
  `DEPS_INDEX_VERSION`, `save_deps_index`, `load_deps_index`.
- `src/deps.rs` and `cargo_metadata` have already been removed from the
  tree — no deletion work is needed there.

**Ticket decisions relevant to this phase (not re-derivable from code):**

- Domain modules (embedder, chunker, search, store, chunk, git, daemon,
  ipc) stay untouched except for the surgical `store.rs` edit below.
- `deps.rs` and `DepsIndexData` are dead weight in the new model; Phase 1
  explicitly removes the serialization format for them (Phase 3 introduces
  a new per-dep index shape under `.debrief/deps/<key>.bin`).
- Single index file contains **merged chunks from all project (`dep = false`)
  sources**. Sources with `dep = true` are skipped with a `tracing::warn!`
  in Phase 1 and picked up in Phase 3.
- Config path: `.debrief/config.toml` at the git root (already returned by
  `config::config_paths().project`).
- Progress reporting during indexing: keep the stderr dots pattern from the
  previous pipeline (`indexing...done. N chunks, M files.`), emitted
  unconditionally. `tracing` spans coexist with it — spans are for
  structured timing, dots are for user visibility.

**Behavior change vs previous pipeline (document in commit, not
user-facing):**

- **Incremental reindex is dropped in Phase 1.** Staleness is a binary
  "commit hash matches last_indexed_commit" check. Any mismatch triggers
  a full rebuild from scratch. This simplification avoids the correctness
  problems that arise when the resolved source set changes between runs
  (adding a source would otherwise need bookkeeping to reindex pre-existing
  files under the new root). Incremental diffs return in a later phase
  once the source set itself has a stable identity.
- `--no-deps` CLI flag and `include_deps` trait parameter are kept for
  signature stability but become no-ops in Phase 1 (logged at `debug`
  via `tracing`). Phase 3 gives them meaning again via the dep-source
  namespace.
- `dep_overview` returns `anyhow::bail!("dependency overview not
  available in Phase 1 of the multi-language sources epic (ticket
  260409); re-enable in Phase 3")` instead of panicking via `todo!()`.

**Data contracts that cross capsule boundaries (flagged per
plan-writer doctrine):**

- `.debrief/config.toml` TOML schema — extended by three new IPC variants
  but shape is owned by `config.rs` and already serialized via serde.
- On-disk index format — bumped from version 7 to 8 because `BackendTag`
  header is retained but `DepsIndexData` is removed. Old indexes return
  `Ok(None)` from `load_index` and trigger a full rebuild silently.
- IPC wire protocol — three new `DaemonRequest` / `DaemonResponse`
  variants (see Step 5). No shape change to existing variants.

## Skeleton Amendments

**Additive — IPC protocol (src/ipc/protocol.rs):**

New `DaemonRequest` variants:

- `AddSource { language: Language, root: PathBuf, dep: bool }`
- `ListSources`
- `RemoveSource { index: usize }`

New `DaemonResponse` variant:

- `Sources { entries: Vec<SourceEntry> }`

`Language` and `SourceEntry` must be re-exported or directly imported
from `crate::config` inside `protocol.rs`. Both already derive
`Serialize`/`Deserialize`.

**Additive — store.rs:**

Bump `INDEX_VERSION` from `7` to `8`. Rationale: `DepsIndexData` is
removed in Phase 1; the on-disk `IndexData` layout itself does not
change, but the version bump is mandated (directive 3) to invalidate
any existing index files co-located with the now-absent deps index and
to guarantee a clean slate for the Phase 1 pipeline semantics.

**Breaking — store.rs:**

- Current contract: `store` exports `DepsIndexData`, `DEPS_INDEX_VERSION`,
  `save_deps_index`, `load_deps_index`, and two test functions
  (`test_deps_index_staleness_check`, `deps_backend_mismatch_returns_none`).
- Proposed: delete all five symbols plus both tests. Keep `IndexData`,
  `BackendTag`, `current_backend`, `save_index`, `load_index` and their
  tests unchanged. (`IndexData::chunks` has no `#[serde(default)]`
  attribute in the current code — do not add one.)
- Rationale: Phase 1 stores a single merged project index; dep sources
  use a different storage shape introduced in Phase 3. Retaining the
  old deps API risks accidental coupling.

**Additive — service.rs pipeline helper (internal, not on the trait):**

A new free function inside `service.rs`:

```rust
async fn run_index_for_sources(
    project_root: &Path,
    sources: &[SourceEntry],
    embedder: &Embedder,
) -> Result<IndexData>
```

Plus a small helper:

```rust
async fn load_or_rebuild_index(
    project_root: &Path,
    embedder: &Embedder,
    force_full: bool,
) -> Result<IndexData>
```

These replace the Phase-0 `run_index` / `run_deps_index` /
`ensure_index_fresh` / `ensure_deps_index_fresh` helpers from the old
service.rs.

## Steps

### Step 1 — store.rs surgical edit (mandatory, directive 3)

Change only these items in `src/store.rs`, without touching `IndexData`
or its existing tests:

- `const INDEX_VERSION: u32 = 7` → `8`.
- Delete `DepsIndexData` struct and its `new`/`Default` impls.
- Delete `DEPS_INDEX_VERSION` constant.
- Delete `save_deps_index` and `load_deps_index` functions.
- Delete `tests::test_deps_index_staleness_check` and
  `tests::deps_backend_mismatch_returns_none`. Leave `sample_chunk`
  usable for the remaining tests.
- Leave the "Dependency index" section header comment deletable (or
  collapse it) — cosmetic.

Do not modify: `BackendTag`, `current_backend`, `IndexData`,
`save_index`, `load_index`, `round_trip`, `missing_file_returns_none`,
`version_mismatch_returns_none`, `backend_mismatch_returns_none`,
`parent_directory_creation`.

### Step 2 — config.rs: implement `append_source` and `remove_source_at`

Contract (already stubbed, now filled):

```rust
pub fn append_source(project_root: &Path, entry: SourceEntry) -> Result<()>;
pub fn remove_source_at(project_root: &Path, index: usize) -> Result<()>;
```

Behavior:

- Resolve the project config path via `config_paths(project_root).project`.
  If `project` is `None` (not inside a git repo), return an error with a
  clear message referencing the git-worktree/submodule limitation from
  `mental-model/config.md`.
- `append_source`: `load_layer_single(&path)?.unwrap_or_default()`,
  push `entry` onto `sources`, `save_config(&path, &config)`. Do not
  merge other layers — writing a single layer directly is the pattern
  documented in `mental-model/config.md`.
- `remove_source_at`: load the project layer (missing = empty). Bounds
  check `index` against `sources.len()`; return
  `anyhow::bail!("source index {index} out of range (have {n} entries)")`
  on overflow. Otherwise `sources.remove(index)` and save.
- Neither function rewrites other fields. `save_config` already handles
  parent directory creation.

**Edge case:** if the resulting `sources` vec is empty after
`remove_source_at`, still write the file with `sources = []`. Do not
delete the config file.

### Step 3 — service.rs: pipeline helpers

Two new private async free functions in `service.rs` (not on the trait):

```rust
async fn load_or_rebuild_index(
    project_root: &Path,
    embedder: &Embedder,
    force_full: bool,
) -> Result<IndexData>;

async fn run_index_for_sources(
    project_root: &Path,
    sources: &[SourceEntry],
    embedder: &Embedder,
) -> Result<IndexData>;
```

Plus a trivial path helper `fn index_path(project_root: &Path) -> Result<PathBuf>`
that returns `<git_root>/.git/debrief/index.bin` using the same
git-root-walk pattern already present in `daemon::daemon_dir`.

**`load_or_rebuild_index` contract:**

1. Compute `index_path(project_root)`.
2. `store::load_index(&path)?` — if `Some(existing)` AND
   `existing.embedding_model == Some(current_model_name)` AND
   `existing.last_indexed_commit == Some(current_head)` AND `!force_full`,
   return `existing`. The model name is whatever the embedder was built
   from; pass it in or capture it at embedder construction.
3. Otherwise call `run_index_for_sources`, set
   `last_indexed_commit = Some(head_commit)` and `embedding_model =
   Some(model_name)` on the returned `IndexData`, `save_index`, and
   return it.
4. If `git::head_commit(project_root)` fails (e.g. empty repo with no
   HEAD), treat staleness as "always rebuild" and carry `None` for
   `last_indexed_commit`.

**`run_index_for_sources` contract:**

1. Wrap top-level body in `tracing::info_span!("rebuild_index", project = %project_root.display())`.
2. Accept `sources: &[SourceEntry]` as a parameter — the caller is
   responsible for resolving the source list (via
   `config::resolve_sources(project_root)?`) and passing it in. Do NOT
   call `config::resolve_sources` inside this helper; that would make
   test injection impossible. `load_or_rebuild_index` must call
   `config::resolve_sources(project_root)?` before calling
   `run_index_for_sources`. Ignore `sources.is_empty()` — return an
   empty `IndexData` with zero counts and a `tracing::warn!("no sources
   registered; rebuild produced empty index")`.
3. Partition: drop `dep == true` entries with `tracing::warn!(root =
   %entry.root.display(), "dep sources are ignored in Phase 1 of ticket
   260409; Phase 3 will implement dependency indexing")`. Phase 1
   operates only on `dep == false` entries.
4. **File discovery stage** — `info_span!("file_discovery")`. For each
   project source entry:
   - Build the scoped pathspec: join `project_root` with `entry.root`
     (canonicalize lexically; accept `"."` as project_root itself).
   - Shell out via the existing `git` helpers. `git::changed_files`
     accepts an `Option<commit>`; Phase 1 always passes `None` inside
     this helper (full rebuild only), which yields
     `git ls-files`-equivalent output over the whole repo. Then in
     Rust, filter the returned paths to: (a) paths under the entry's
     root directory (prefix match with path separator handling), and
     (b) paths whose extension is in the effective extension set:
     `entry.extensions` if `Some`, else the language default
     (`Language::Rust → ["rs"]`, `Language::Cpp → ["cpp", "cc", "cxx",
     "c", "h", "hpp", "hxx", "hh"]`).
   - Deduplicate across overlapping source roots by keeping a
     `HashSet<(relative_path, Language)>`. If the same file matches two
     entries with **different** languages, pick the one registered
     first and `tracing::warn!` on conflict. Same-language duplicates
     are silently unified.
5. **Chunking stage** — `info_span!("chunking", files = files.len())`.
   For each `(file, language)`:
   - Read the file from `project_root.join(&relative_path)`.
   - `let chunker = chunker_for(&language);`
   - `chunker.chunk(&relative_path, &contents)` — pass relative path so
     `RustChunker::derive_module_path` works correctly
     (`mental-model/chunker.md` common mistake).
   - Accumulate into a `HashMap<PathBuf, Vec<Chunk>>` keyed by relative
     path, matching `IndexData::chunks` shape.
   - On per-file error: log `tracing::warn!` and continue — do not
     abort the whole rebuild for a single broken file.
6. **Embedding stage** — `info_span!("embedding", chunks = total)`.
   - Flatten all chunks into a `Vec<&mut Chunk>` preserving map
     references.
   - Batch by `EMBED_BATCH_SIZE = 64`: for each batch, collect the
     `embedding_text`s, call `embedder.embed_batch(&texts)?`, assign
     the returned vectors back into `chunk.embedding = Some(vec)`.
   - Progress: preserve the previous stderr idiom — print `indexing`
     once, `.` per batch (flush stderr), then `\ndone. N chunks, M
     files.` at the end. The `tracing` span wraps this block; the
     stderr output is user-facing and independent of `RUST_LOG`.
7. **Index build stage** — `info_span!("index_build")`. Construct
   the result via `IndexData::new()` (the `version` and `backend`
   fields are **private** and cannot be set via struct literal from
   outside `store.rs`), then assign `data.chunks = map`. Return `data`.
   `last_indexed_commit` and `embedding_model` are set by the caller
   (`load_or_rebuild_index`) on the returned value, because it owns the
   decision about which commit/model to record.

### Step 4 — service.rs: InProcessService method bodies

Fill the eight method bodies on `InProcessService`:

- **`index(project_root, _path, _include_deps)`** — Phase 1 ignores
  both `path` and `include_deps`. Resolve the embedding model from
  config; construct `Embedder` once; call `load_or_rebuild_index(…,
  force_full = true)`; count files and chunks from the returned
  `IndexData`; return `IndexResult { files_indexed, chunks_created }`.
- **`search(project_root, query, top_k, _include_deps)`** —
  construct `Embedder`; `load_or_rebuild_index(…, force_full = false)`;
  flatten chunks into a `Vec<(PathBuf, Chunk)>`; build a fresh
  `SearchIndex`; `search_index.search(query, &embedder, top_k)?`
  (argument order: query, embedder, top_k — matches the actual
  `SearchIndex::search` signature);
  return the result vec. `include_deps` is a no-op in Phase 1
  (log at `tracing::debug!`).
- **`overview(project_root, file)`** — construct `Embedder`;
  `load_or_rebuild_index(…, force_full = false)`; fetch
  `data.chunks.get(&file)` (normalize `file` to relative first); if
  missing, `anyhow::bail!` with "no index entries for {file}"; format
  chunks into a human-readable skeleton string. Format can mirror
  whatever the previous implementation produced — "sort by line_range,
  join display_text with blank lines" is acceptable. Keep the output
  single-column plain text.
- **`dep_overview(_project_root, _crate_name)`** — `anyhow::bail!`
  with "dependency overview not yet available (Phase 3 of ticket
  260409)". Do NOT `todo!()` — `bail!` is non-panicking and makes the
  CLI exit cleanly.
- **`set_embedding_model(project_root, model, global)`** — validate
  via `ModelRegistry::lookup(model).is_some()`; reject unknown models
  with `anyhow::bail!`. Write to the global or project config layer
  following the `load_layer_single → mutate → save_config` pattern
  documented in `mental-model/service.md`. This is the reference
  implementation pattern.
- **`add_source`** — already implemented in the skeleton, delegates
  to `config::append_source`. Once `append_source` (Step 2) is real,
  this works without further change.
- **`list_sources`** — already implemented. Unchanged.
- **`remove_source`** — already implemented, delegates to
  `config::remove_source_at`. Once `remove_source_at` is real, works.

**Embedder construction detail:** introduce a small private helper
`fn build_embedder(project_root: &Path) -> Result<(Embedder, String)>`
that loads config, resolves the model name (falling back to the
registry's default), calls `ModelRegistry::lookup`, ensures model
files, constructs the `Embedder`, and returns it plus the canonical
model name string. Every service method that needs an embedder calls
this helper — do not inline the logic.

### Step 5 — ipc/protocol.rs + daemon.rs + DaemonClient wiring

Three legs, following the recipe in `mental-model/daemon.md`:

1. Add the four new variants to `DaemonRequest` / `DaemonResponse` (see
   Skeleton Amendments). Import `Language` and `SourceEntry` from
   `crate::config`. Add roundtrip tests for each new request variant
   and for the `Sources` response, matching the shape of existing
   `roundtrip_*` tests.
2. Add dispatch arms in `daemon::handle_request` for the three new
   requests, each following the existing pattern (match service result,
   wrap error in `DaemonResponse::Error`). `ListSources` returns
   `DaemonResponse::Sources { entries }`; `AddSource` and `RemoveSource`
   return `DaemonResponse::Ok { message }` with a one-line confirmation.
3. Implement the three methods on `DaemonClient` — call `self.send(…)`,
   match on expected response variants, return `anyhow::bail!` on
   `Error` or unexpected variant. The existing methods (`index`,
   `search`, etc.) are still `todo!()` in the skeleton — fill those
   too, using the same send/match pattern. Unknown daemon response →
   `anyhow::bail!("unexpected response from daemon: {other:?}")`.

**Service dispatch pattern:** the `Service` struct (which wraps
`Option<DaemonClient>` and falls back on daemon error) already has
arms for all eight trait methods in the skeleton. No changes needed
there.

### Step 6 — main.rs adjustments

Skeleton is nearly complete; only minor tweaks needed:

  Phase 1, either silently ignore or emit a `tracing::debug!`. Still

- The existing `Sources` and `Remove` dispatch arms already print the

- Nothing else requires editing in `main.rs`.

### Step 7 — integration test additions

The existing `tests/integration_multi_language_sources.rs` already
covers (a) backward-compat `list_sources` and (b) `chunker_for(Rust)`
dispatch. Phase 1 adds:

- **`rebuild_index_with_explicit_rust_source`**: build a tempdir,
  `git init`, plant a Cargo.toml plus one or two `.rs` files under
  `src/`, write a `.debrief/config.toml` with a single Rust source at
  root, call `service.index(root, root, false)`, assert
  `result.chunks_created > 0` and `result.files_indexed ≥ 1`. Then
  re-run `index` and assert the second call also succeeds (full
  rebuild is idempotent).
- **`rebuild_index_skips_dep_sources_in_phase1`**: plant a config with
  one `dep = true` source and one `dep = false` source. Assert the
  resulting index count matches what the non-dep source alone produces
  — dep sources are silently dropped. This pins the Phase 1 semantics
  that Phase 3 later flips.
- **`add_then_list_then_remove_roundtrip`**: starting from an empty
  git repo (no Cargo.toml), call `add_source(… Cpp, "src/", false)`,
  then `list_sources` — expect one entry with the right language.
  Call `remove_source(0)` — expect next `list_sources` to return empty.
- **`rebuild_index_unknown_language_is_isolated`** (optional): with a
  Cpp source registered, the current `CppChunker` stub panics via
  `todo!()`. Either guard the rebuild test so it never actually chunks
  a `.cpp` file (use an empty directory so file discovery returns
  zero files), or mark the Cpp test with `#[ignore]` until Phase 2
  completes. Recommended: use an empty directory so the test exercises
  the skip path without triggering the stub.

Tests should set `CARGO_DEBRIEF_NO_DAEMON=1` via
`InProcessService::new()` directly to bypass daemon spawning. The
embedder is heavyweight — these tests take several seconds because
each run downloads the model on a cold cache. That is acceptable; no
test-mode embedder stub is in scope for Phase 1.

## Testing

- **TDD**: Step 2 (config append/remove) and Step 5 (IPC roundtrip)
  are small, pure, and benefit from tests-first. Write the roundtrip
  tests and the `remove_source_at` bounds-check test before bodies.
- **Post-impl**: Steps 3, 4, 7 — the pipeline is too intertwined with
  embedder/chunker/filesystem to TDD productively. Write the
  integration tests after the pipeline compiles end-to-end.
- **Manual**: smoke-run `cargo debrief rebuild-index` and `cargo debrief
  search "Chunker"` on this repository; verify:
  - The rebuild completes without panic.
  - `RUST_LOG=info cargo debrief rebuild-index` emits the four expected
    spans (`file_discovery`, `chunking`, `embedding`, `index_build`)
    with timings.
  - `cargo debrief sources` reports one auto-detected Rust source
    before any `add`, and the explicit list after.
  - Search returns project hits with non-zero scores.

## Success Criteria

- `cargo check --all-targets` passes on default features (`wgpu`) and
  on `--no-default-features --features ort-cpu`.
- `cargo clippy --all-targets -- -D warnings` is clean.
- `cargo test` — all existing tests still pass; new
  `integration_multi_language_sources` tests pass.
- `cargo debrief rebuild-index` on this repo indexes all `.rs` files
  under the repo root and writes a fresh `.git/debrief/index.bin` with
  `version = 8`.
- `cargo debrief search "DebriefService"` returns at least one hit
  from `src/service.rs`.
- `cargo debrief add cpp src --no-dep` then `cargo debrief sources`
  shows a Cpp entry; `cargo debrief remove 0` clears it.
- With `RUST_LOG=info`, every rebuild emits exactly the four span
  names listed in directive 5 — verified by reading stderr manually.
- No `deps-index.bin` path is written or read anywhere in the Phase 1
  code. `DepsIndexData`, `save_deps_index`, `load_deps_index`,
  `DEPS_INDEX_VERSION` symbols are absent from `store.rs`.

  by using empty directories for any Cpp-source test fixture).
