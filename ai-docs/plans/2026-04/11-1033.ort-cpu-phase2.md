# ort CPU Phase 2 — ONNX Inference Path

## Context

This is Phase 2 of `260411-refactor-ort-cpu-revival`. Phase 1 already landed
the Cargo feature wiring, `compile_error!` mutual-exclusion guards in
`src/lib.rs`, and a stub `#[cfg(feature = "ort-cpu")] Embedder` with
`unimplemented!()` bodies in `src/embedder.rs`. The `ort = "=2.0.0-rc.12"`
dep is already declared with the `download-binaries` feature. The tokenizer
import (`tokenizers::Tokenizer`) is already gated on
`any(feature = "wgpu", feature = "ort-cpu")`.

Phase 2 replaces the stubs with a working ONNX inference path that produces
identical-shape output (`Vec<Vec<f32>>`, each inner vec 768 floats, L2
normalized) to the wgpu path, using CPU EP only.

### Ticket-mandated constraints (do not re-derive)

- **CPU EP only** — no CoreML, no CUDA, no DirectML, no automatic EP detection.
- **Tokenizer: reuse the existing `tokenizers` crate** — same `tokenizer.json`
  file as wgpu. Do not switch to ort's built-in tokenizer.
- **Cache layout migration** — `<cache>/<model-name>/` becomes
  `<cache>/<model-name>/burn/` and `<cache>/<model-name>/ort/`. Old flat
  safetensors caches are orphaned by design; no migration code.
- **ONNX output contract** — the model outputs a tensor named
  `last_hidden_state` with shape `[batch_size, seq_len, 768]`. Pooling is
  attention-mask-weighted mean over seq_len, then L2 normalize. This is the
  same algorithm `burn_mean_pooling` + `burn_l2_normalize` implement in the
  wgpu path.
- **Scope** — Phase 2 is inference + cache layout only. Phase 3 handles the
  index backend tag. Phase 4 removes NdArray.

### Research-discovered facts

- **ort 2.0.0-rc.12 API vs directive terminology.** The plan directive
  references `CpuExecutionProvider::default()`. The actual rc.12 symbol is
  `ort::ep::CPU` (not `CpuExecutionProvider`). Construction is
  `ort::ep::CPU::default().build()` returning an
  `ExecutionProviderDispatch`. `with_execution_providers` takes
  `impl AsRef<[ExecutionProviderDispatch]>`. Intent of the directive —
  register CPU EP via `SessionBuilder::with_execution_providers([...])`
  using no-config defaults — is preserved.
- **HuggingFace ONNX artifacts** for `nomic-ai/nomic-embed-text-v1.5` live
  under the `onnx/` subdirectory. CPU fp32 inference uses `onnx/model.onnx`
  (547 MB, full precision). Other variants (`model_fp16.onnx`,
  `model_int8.onnx`, `model_q4.onnx`, etc.) exist but are out of scope.
  `config.json` and `tokenizer.json` remain at the repo root — they are
  identical between burn and ort paths but must be downloaded into the
  per-backend subdir for self-containedness (ticket D4).
- **No `model.onnx_data` sidecar** exists in the onnx/ directory for the
  fp32 model — `model.onnx` is self-contained.
- **Session input names.** nomic-embed-text-v1.5 is a BERT-style encoder;
  standard ONNX export uses input names `input_ids`, `attention_mask`, and
  (optionally) `token_type_ids`. The executor must verify actual names via
  `session.inputs()` after first load and adjust the feed if the export
  uses different names. Do not hardcode without this verification step.
- **Tensor data types.** ONNX BERT exports typically use `int64` for
  `input_ids` / `attention_mask`. The wgpu path widens `u32 → i64` via
  `token_ids_to_burn_tensor`. The ort path must build `i64` arrays directly.
- **Output extraction.** Use `Tensor::extract_tensor()` which returns
  `(&Shape, &[f32])`. This avoids pulling in an `ndarray` dependency for
  just the pooling step — manual index arithmetic is simple enough for 3D.
- **`download-binaries` feature** is already enabled on the `ort` dep, so
  no system-installed ONNX Runtime is required — the statically linked
  binary is downloaded at build time by the ort crate's build script.

## Skeleton Amendments

None breaking. Clarification only:

- **Terminology clarification (non-breaking).** The implementation uses
  `ort::ep::CPU::default().build()` in place of the directive's
  `CpuExecutionProvider::default()`. Same semantic — no-config CPU EP via
  `with_execution_providers` — different symbol name in rc.12. Public
  `Embedder::{load, embed_batch, embed}` signatures are untouched.

- **`ensure_model_files` signature stays the same** `(spec, cache_dir) ->
  Result<(PathBuf, PathBuf, PathBuf)>`. The tuple semantics
  `(weights, config, tokenizer)` are preserved; under `ort-cpu` the first
  element is a `.onnx` path instead of a `.safetensors` path. Both callers
  (wgpu `Embedder::load` and the new ort-cpu `Embedder::load`) pattern-match
  the same tuple shape.

- **`ModelSpec::weights_path` is no longer authoritative under `ort-cpu`.**
  The field still exists (it is `pub` and referenced by the `ModelRegistry`
  unit tests) and still holds `"model.safetensors"`. The ort-cpu code path
  ignores this field and uses a feature-gated constant for the ONNX
  filename. This is a deliberate local inconsistency — adding a second
  field just to avoid it is not worth the churn given the field is used as
  a URL segment only, and `ModelRegistry::lookup` users care about `name`,
  `max_length`, and `embedding_dim` only.

## Steps

### Step 1 — Cache layout migration in `ensure_model_files`

Rewrite `ensure_model_files` so the effective model directory is
`cache_dir.join(spec.name).join(<backend>)`, where `<backend>` is the
compile-time constant `"burn"` under `#[cfg(feature = "wgpu")]` and
`"ort"` under `#[cfg(feature = "ort-cpu")]`.

Use one `#[cfg(feature = "wgpu")]` / `#[cfg(feature = "ort-cpu")]` split at
the top of the function body to pick:

1. The subdir name (`"burn"` vs `"ort"`).
2. The weights destination filename (`"model.safetensors"` vs `"model.onnx"`).
3. The weights source URL segment within the HF repo:
   - wgpu: `spec.weights_path` (currently `"model.safetensors"`)
   - ort-cpu: the constant `"onnx/model.onnx"`

`config.json` and `tokenizer.json` logic is unchanged — same filenames, same
`spec.config_path` / `spec.tokenizer_path` source paths, just written under
the new per-backend subdir.

Remove the `#[allow(dead_code)]` on `ensure_model_files` and `download_file`
— both become live under `ort-cpu` in this phase.

Keep the function returning `(weights_dest, config_dest, tokenizer_dest)`.
The wgpu `Embedder::load` call site already consumes the tuple in this
order; the ort-cpu `Embedder::load` will do the same.

**Orphan cleanup:** do nothing. Old flat `<cache>/<model-name>/model.safetensors`
files are intentionally orphaned (ticket Non-goals #5). No warning log, no
migration utility.

**Concurrent-download race:** the existing `download_file` handles the
`dest.exists()` race by discarding its `.tmp` copy. Do not change that logic.

### Step 2 — Feature-gated imports and fields

At the top of `src/embedder.rs`, under `#[cfg(feature = "ort-cpu")]`:

- Import `ort::{session::Session, session::builder::GraphOptimizationLevel,
  value::Tensor, ep}`. Concrete paths may need minor adjustment based on
  rc.12 re-exports — the executor should compile-check and resolve via
  rust-analyzer if the above don't resolve verbatim.
- **Do not import `inputs` via `use`** — `ort::inputs!` is a `#[macro_export]`
  macro; in Rust 2024 edition it is accessed as `ort::inputs![...]` directly
  without a `use` statement. Importing it as `use ort::inputs` will fail
  unless ort re-exports it via `pub use` (verify; if it does work, the
  import is fine).
- Import `tokenizers::{PaddingParams, PaddingStrategy, TruncationParams,
  TruncationStrategy}` — the same truncation/padding configuration as the
  wgpu path is needed. Widen the existing `#[cfg(feature = "wgpu")] use
  tokenizers::{...}` line to `#[cfg(any(feature = "wgpu", feature = "ort-cpu"))]`.

Augment the `#[cfg(feature = "ort-cpu")] pub struct Embedder`:

```rust
#[cfg(feature = "ort-cpu")]
pub struct Embedder {
    session: std::sync::Mutex<ort::session::Session>,
    tokenizer: Tokenizer,
    max_length: usize,
}
```

Rationale for `Mutex<Session>`: `ort::Session::run` takes `&mut self` in
rc.12, and we already serialize the wgpu path via `Mutex<BurnNomicBertModel>`.
Matching that pattern keeps `Embedder: Send + Sync` and the public API
unchanged. Remove the `#[allow(dead_code)]` attributes on `tokenizer` and
`max_length`; both become live.

### Step 3 — Implement `Embedder::load` (ort-cpu)

```rust
#[cfg(feature = "ort-cpu")]
impl Embedder {
    pub async fn load(model_name: &str, cache_dir: &Path) -> Result<Self> { ... }
}
```

Body contract:

1. `ModelRegistry::lookup(model_name)` — same as wgpu path.
2. `ensure_model_files(&spec, cache_dir).await?` — returns
   `(onnx_file, config_file, tokenizer_file)` under the new `ort/` subdir.
3. Build a `Session`:
   ```rust
   let cpu_ep = ort::ep::CPU::default().build();
   let session = ort::session::Session::builder()?
       .with_execution_providers([cpu_ep])?
       .with_optimization_level(GraphOptimizationLevel::Level3)?
       .commit_from_file(&onnx_file)?;
   ```
   - `Level3` is ort's most aggressive graph optimization level and is
     safe for static-shape encoder inference. If it fails at commit time
     on some model, drop to `Level1`.
   - Do **not** call `with_intra_threads` / `with_inter_threads` — let ort
     default to the runtime's CPU count. No directive to tune thread pool.
4. Load the tokenizer from `tokenizer_file` via `Tokenizer::from_file`
   and apply the exact same `with_truncation(Some(TruncationParams { max_length,
   strategy: LongestFirst, .. }))` and
   `with_padding(Some(PaddingParams { strategy: BatchLongest, .. }))` as
   the wgpu path. This is a copy-paste of the wgpu block.
5. Return `Self { session: Mutex::new(session), tokenizer, max_length:
   spec.max_length }`.

`config_file` is read but **not parsed** in this path — it is downloaded
for self-containedness only. The wgpu path parses it for `NomicBertConfig`;
the ort path uses the ONNX graph directly and does not need the JSON. Do
not read or parse `config.json` in `load()` — skip it. (If later debugging
shows it is needed for e.g. validation, add a read at that time.)

### Step 4 — Implement `Embedder::embed_batch` (ort-cpu)

```rust
#[cfg(feature = "ort-cpu")]
impl Embedder {
    pub fn embed_batch(&self, texts: &[&str]) -> Result<Vec<Vec<f32>>> { ... }
}
```

Body contract:

1. Empty input guard: `if texts.is_empty() { return Ok(vec![]); }` — same
   as wgpu path.
2. `self.tokenizer.encode_batch(texts.to_vec(), true)` — identical to wgpu.
3. Compute `batch_size` and `seq_len` the same way
   (`max len clamped to self.max_length`).
4. Build two `Vec<i64>` of length `batch_size * seq_len`:
   - `input_ids_data` — copy `enc.get_ids()` (u32 → i64 cast).
   - `attention_mask_data` — copy `enc.get_attention_mask()` (u32 → i64).
   Layout is row-major `[batch, seq]`.
5. Wrap as ort tensors and run:
   ```rust
   // Shape element type: ort rc.12 tuple API typically uses i64 (ONNX convention).
   // If from_array rejects i64, try [batch_size, seq_len] (usize) or
   // ndarray::IxDyn — resolve via rust-analyzer.
   let shape = [batch_size as i64, seq_len as i64];
   let input_ids = ort::value::Tensor::from_array((shape, input_ids_data))?;
   let attention_mask = ort::value::Tensor::from_array((shape, attention_mask_data.clone()))?;
   let mut session = self.session.lock().unwrap();
   let outputs = session.run(ort::inputs![
       "input_ids" => input_ids,
       "attention_mask" => attention_mask,
   ])?;
   ```
   - The `ort::inputs!` macro accepts `"name" => Tensor` pairs and returns
     a `SessionInputs` value consumed by `session.run`.
   - **Executor-verified step:** before trusting the names `"input_ids"`
     and `"attention_mask"`, the executor must run the ignored integration
     test once and observe the result. If `session.run` fails with
     "missing input" or the model actually expects different names, fix
     the literals. BERT-style ONNX exports almost always use these names,
     but verify. If `token_type_ids` is also a required input, feed a
     zero-filled `i64` tensor of the same shape (the wgpu path ignores
     token type IDs because burn's NomicBert implementation doesn't use
     them, and zeros are the "single sentence" convention).
6. Extract `last_hidden_state` — **pseudo-code only; resolve against live
   crate before writing**:
   ```rust
   // Pseudo-code — do NOT copy verbatim; method names require verification.
   // In ort rc.12, session.run() returns SessionOutputs<'_>.
   // Indexing by string (outputs["last_hidden_state"]) yields a DynValue.
   // try_extract_tensor::<f32>() on DynValue returns Result<TensorView<'_, f32, IxDyn>>,
   // NOT a tuple — destructuring as (shape, data) will not compile.
   //
   // Correct pattern (verify method names with rust-analyzer):
   let output_val = &outputs["last_hidden_state"];
   let tensor_view = output_val.try_extract_tensor::<f32>()?;
   let shape = tensor_view.shape();   // &[usize] — expect [batch, seq, hidden]
   let data: &[f32] = tensor_view.as_slice().expect("contiguous f32 tensor");
   ```
   - `outputs` may be indexed by `&str` (string literal) or by integer
     position — prefer string to be robust against output ordering changes.
   - If the ort rc.12 API does not expose `as_slice()`, try `view()` →
     `as_slice_memory_order()` or use raw pointer extraction. The contract
     is: obtain a shape `[batch, seq, 768]` and a contiguous f32 slice of
     length `batch * seq * 768`. Verify with `shape[0] == batch_size`,
     `shape[1] == seq_len`, `shape[2] == embedding_dim` via debug assert.
   - **Do not add an `ndarray` dependency** — use only the slice and shape
     values; all pooling is done via index arithmetic in step 7.
7. **Mean pooling over seq dimension, weighted by attention mask:**

   For each batch index `b` in `0..batch_size`:
   - Initialize `sum: [f32; 768]` to zeros, `weight_sum: f32 = 0.0`.
   - For each seq index `s` in `0..seq_len`:
     - If `attention_mask_data[b * seq_len + s] == 0`, skip.
     - Else, `weight_sum += 1.0`, and for each hidden index `h in 0..768`,
       `sum[h] += data[b * seq_len * 768 + s * 768 + h]`.
   - If `weight_sum == 0.0`, treat as a clamp to `1e-9` (defensive; empty
     attention masks shouldn't happen but don't panic).
   - Divide `sum[h] /= weight_sum` for `h in 0..768`.
8. **L2 normalize** each pooled `[f32; 768]`:
   - `norm = (sum_h sum[h]^2).sqrt().max(1e-12)`
   - `sum[h] /= norm`
9. Push `sum.to_vec()` to the result. Return `Ok(result)`.

Use `spec.embedding_dim` (768) via a local constant or by threading the
embedding_dim through the returned shape. Do **not** hardcode `768` in
more than one place inside this function — take it from the output shape.

### Step 5 — `Embedder::embed` (ort-cpu)

No change — the existing stub body is correct:

```rust
pub fn embed(&self, text: &str) -> Result<Vec<f32>> {
    let mut batch = self.embed_batch(&[text])?;
    batch.pop().context("embed_batch returned empty result for single text")
}
```

Remove the stub's `unimplemented!` from `embed_batch` (Step 4) and the
`embed` body will just work.

### Step 6 — Re-enable / un-gate shared utilities

- Remove `#[allow(dead_code)]` from `ensure_model_files` and `download_file`.
- Keep the existing `#[cfg(feature = "wgpu")] fn get_burn_device` — it is
  wgpu-only and must stay feature-gated. Do not introduce an ort-side
  analogue (there is no device selection on the ort path).

### Step 7 — Compile matrix verification

The executor must verify before declaring the phase done:

- `cargo check --no-default-features --features ort-cpu` — must pass with
  no warnings beyond those present on master.
- `cargo check` (default wgpu) — must still pass; the wgpu path must be
  untouched in behavior.
- `cargo check --features ort-cpu` — must fail with the
  `compile_error!("features ... are mutually exclusive ...")` from `lib.rs`.
- `cargo check --no-default-features` — must fail with the "one of features
  ... must be enabled" `compile_error!`.

## Testing

### Unit tests (post-impl)

The existing `#[ignore]`-gated tests `embed_batch_returns_normalized_vectors`
and `embed_batch_truncates_long_input` become the acceptance harness for
the ort path as well. They already call `Embedder::load` and
`embed_batch` via the public API, so they are backend-agnostic.

- Run under `cargo test --no-default-features --features ort-cpu -- --ignored
  embed_batch_returns_normalized_vectors`. Expected: passes with 768-dim
  unit-norm vectors (`norm` within `1e-5` of `1.0`).
- Run `embed_batch_truncates_long_input`. Expected: `embed(&long_text)` on
  ~600 tokens returns `Ok` with a valid vector (no panic, no error).

### Manual smoke (required before reporting done)

From the ticket Phase 2 success criteria: "produces embeddings of correct
shape and approximate magnitude". The executor should:

1. Delete or move aside `<data_dir>/debrief/models/nomic-embed-text-v1.5/`
   if it exists, to force a fresh download.
2. `cargo run --no-default-features --features ort-cpu -- rebuild-index`
   on a small test project (e.g. `test-repos/ripgrep` if present, or any
   small crate with ~10 files).
3. Verify the cache layout `<data_dir>/debrief/models/nomic-embed-text-v1.5/ort/{model.onnx,config.json,tokenizer.json}`
   is created.
4. Run a sample `search` query and confirm results come back with non-zero
   distances. The cross-backend equivalence of embeddings is explicitly
   out of scope (ticket Non-goals #4) — only that results are non-garbage.

### What is NOT tested in Phase 2

- Cross-backend index invalidation (Phase 3 work).
- Wall-time throughput matching the historical 9m37s baseline (Phase 5
  validation on the full `test-repos/ripgrep` workload).
- Coexistence of `burn/` and `ort/` subdirs with both binaries (Phase 5).

## Success Criteria

Observable conditions, all must hold:

1. `cargo build --no-default-features --features ort-cpu` succeeds with
   no compile errors or warnings beyond master baseline.
2. `cargo build` (default wgpu) still succeeds unchanged.
3. The ort-cpu binary, when run against a small repo, downloads ONNX files
   into `<cache>/<model-name>/ort/` (not the flat old layout) and produces
   a working embedding index.
4. `embed_batch_returns_normalized_vectors` passes on the ort-cpu build:
   output is `Vec<Vec<f32>>` with each inner vec of length 768 and L2 norm
   within `1e-5` of 1.0.
5. `embed_batch_truncates_long_input` passes on the ort-cpu build: a
   600-token input returns `Ok` without panic.
6. The wgpu build continues to write to `<cache>/<model-name>/burn/`
   (migrated layout) on a fresh cache. Existing flat-layout caches are
   orphaned and ignored — no error, no migration, no warning.
7. `ModelRegistry::lookup` unit tests still pass on both feature configs
   (they only inspect `ModelSpec` fields, unaffected by the inference path).
