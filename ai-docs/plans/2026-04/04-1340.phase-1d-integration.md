# Phase 1D — End-to-End Integration

## Context

Three `InProcessService` methods are stubs returning `anyhow::bail!`:
`index`, `search`, `overview`. Phase 1D wires them together using the
already-complete modules: `git`, `chunker`, `embedder`, `search`, `store`.

### Key decisions from ticket discussion (2026-04-04)

- `search` and `overview` auto-check index freshness before executing
  (implicit indexing). A shared helper `ensure_index_fresh` handles this
  and is called at the start of both methods.
- `rebuild-index` CLI always forces a full reindex (passes `None` as the
  prior commit to `changed_files`).
- The `path` argument on the `index` trait method is vestigial in Phase 1.
  `project_root` is the authoritative base for all git/config/index I/O.
  The executor should ignore `path` and use `project_root` for everything.
- Only `.rs` files are indexed for MVP (filter before chunking).
- `dirs::data_dir()` returning `None` must produce a descriptive error,
  not a panic.

### IndexData already has `embedding_model: Option<String>`

`store::IndexData` already carries both `last_indexed_commit` and
`embedding_model` — no new fields or `INDEX_VERSION` bump needed.

## Steps

### Step 1 — Add a private `index_path` helper in `service.rs`

Computes `.git/debrief/index.bin` relative to the git root.

```rust
fn index_path(project_root: &Path) -> Result<PathBuf> {
    // Walk parents looking for .git directory — mirrors find_git_root logic
    // in config.rs. Cannot call find_git_root because it is private.
    // Acceptable duplication for MVP; refactor if a third caller appears.
    let mut current = project_root;
    loop {
        let candidate = current.join(".git");
        if candidate.is_dir() {
            return Ok(candidate.join("debrief").join("index.bin"));
        }
        current = current.parent()
            .context("not inside a git repository; cannot locate index path")?;
    }
}
```

### Step 2 — Add a private `make_embedder` helper in `service.rs`

Constructs an `Embedder` from config, handling `dirs::data_dir() == None`.

```rust
async fn make_embedder(model_name: &str) -> Result<Embedder> {
    let cache_dir = dirs::data_dir()
        .context("cannot determine user data directory (no home directory?)")?
        .join("debrief")
        .join("models");
    Embedder::load(model_name, &cache_dir).await
}
```

### Step 3 — Add a private `ensure_index_fresh` helper in `service.rs`

Compares HEAD commit and configured model name against the stored index.
Performs full or incremental reindex as needed, then returns the up-to-date
`IndexData`.

```rust
async fn ensure_index_fresh(project_root: &Path) -> Result<IndexData>
```

Logic:

1. Resolve config: `config_paths(project_root)` → `load_config(&paths)`
2. Resolve model name: `config.embedding_model.as_deref().unwrap_or(ModelRegistry::DEFAULT_MODEL)`
3. Resolve index path: `index_path(project_root)?`
4. Load existing index: `store::load_index(&idx_path)?` → `Option<IndexData>`
5. Get current HEAD: `git::head_commit(project_root)?`
6. Decide mode:
   - `None` index → full reindex
   - model name in index != resolved model name → full reindex
   - `last_indexed_commit != Some(head)` → incremental reindex
   - otherwise → return existing index unchanged
7. Call the shared `run_index` function (Step 4) with the determined mode
8. Return the freshly saved `IndexData`

No user-visible output from this helper (it is called silently before
search/overview).

### Step 4 — Implement a private `run_index` function in `service.rs`

```rust
async fn run_index(
    project_root: &Path,
    prior_commit: Option<&str>,     // None = full reindex; Some(hash) = incremental
    existing_index: Option<IndexData>,
) -> Result<(IndexData, IndexResult)>
```

This function contains all the actual pipeline logic and is called by both
`InProcessService::index` (always `prior_commit = None`) and
`ensure_index_fresh` (incremental or full depending on freshness check).

**Pipeline:**

1. Resolve config and model name (same as Step 3, steps 1–2).
2. Build embedder: `make_embedder(model_name).await?`
3. Determine changed files: `git::changed_files(project_root, prior_commit)?`
   - `prior_commit = None` → returns all tracked files via `git ls-files`
   - `prior_commit = Some(hash)` → diffs `hash..HEAD`
4. Start from `existing_index.unwrap_or_else(IndexData::new)` as the mutable base.
5. Remove deleted files: for each path in `changes.deleted`, remove the entry
   from `base_index.chunks`.
6. Collect files to embed: chain `changes.added` + `changes.modified`, filter
   to paths ending in `.rs`.
7. For each `.rs` file to embed:
   a. Read source: `std::fs::read_to_string(project_root.join(&rel_path))?`
   b. Chunk: `RustChunker.chunk(Path::new(&rel_path), &source)?`
      — `rel_path` is already repo-relative from `git::changed_files`; pass as
      `Path::new(&rel_path)` to satisfy `RustChunker`'s relative-path requirement.
   c. Collect `embedding_text` strings for batch embedding.
8. Embed all collected texts in one call:
   `embedder.embed_batch(&texts.iter().map(|s| s.as_str()).collect::<Vec<_>>())?`
   — `embed_batch` is synchronous; call with `tokio::task::spawn_blocking` to
   avoid blocking the async runtime for large batches. See constraint below.
9. Assign embeddings back to chunks (by index position).
10. Insert updated chunks into `base_index.chunks`, keyed by `PathBuf::from(&rel_path)`.
11. Set `base_index.last_indexed_commit = Some(git::head_commit(project_root)?)`.
12. Set `base_index.embedding_model = Some(model_name.to_string())`.
13. Save: `store::save_index(&idx_path, &base_index)?`
14. Return `(base_index, IndexResult { files_indexed, chunks_created })`.

**spawn_blocking constraint:**
`Embedder::embed_batch` holds a `Mutex<Session>` and is CPU-bound. Calling it
directly in an async function works but blocks the tokio thread. Use
`tokio::task::spawn_blocking` to move the inference call to the blocking pool.
Because `Embedder` is not `Send` (due to the ONNX session), wrap only the
`embed_batch` invocation — do not move the whole embedder into the closure.
The simplest correct approach: call `embed_batch` synchronously inside the
`async fn` (acceptable for Phase 1, noted as tech debt), or wrap per-call.
The plan permits the synchronous approach for Phase 1 given single-user CLI use.

### Step 5 — Implement `InProcessService::index`

```rust
async fn index(&self, project_root: &Path, _path: &Path) -> Result<IndexResult> {
    // Full reindex: no prior commit, no existing index.
    let (_data, result) = run_index(project_root, None, None).await?;
    Ok(result)
}
```

The `_path` argument is intentionally unused (see Context above).

### Step 6 — Implement `InProcessService::search`

```rust
async fn search(&self, project_root: &Path, query: &str, top_k: usize) -> Result<Vec<SearchResult>> {
    let index_data = ensure_index_fresh(project_root).await?;
    let config = load_config(&config_paths(project_root))?;
    let model_name = config.embedding_model.as_deref()
        .unwrap_or(ModelRegistry::DEFAULT_MODEL);
    let embedder = make_embedder(model_name).await?;
    let flat_chunks: Vec<(PathBuf, Chunk)> = index_data.chunks
        .into_iter()
        .flat_map(|(path, chunks)| {
            chunks.into_iter().map(move |c| (path.clone(), c))
        })
        .collect();
    let search_index = SearchIndex::build(flat_chunks)?;
    search_index.search(query, &embedder, top_k)
}
```

### Step 7 — Implement `InProcessService::overview`

```rust
async fn overview(&self, project_root: &Path, file: &Path) -> Result<String> {
    let index_data = ensure_index_fresh(project_root).await?;
    // Normalize `file` to a repo-relative string for key lookup.
    // The index is keyed by the path as returned by git (repo-relative).
    let file_key = PathBuf::from(file);
    let chunks = index_data.chunks.get(&file_key)
        .ok_or_else(|| anyhow::anyhow!("no index entry for {}", file.display()))?;
    let overview_text: String = chunks.iter()
        .filter(|c| c.metadata.chunk_type == ChunkType::Overview)
        .map(|c| c.display_text.as_str())
        .collect::<Vec<_>>()
        .join("\n\n");
    if overview_text.is_empty() {
        anyhow::bail!("no overview chunks found for {}", file.display());
    }
    Ok(overview_text)
}
```

**File key normalization caveat:** The index key is whatever path string
`git::changed_files` returned (repo-relative, e.g., `src/main.rs`). The CLI
passes `file` from user input, which may be absolute or relative to cwd.
The executor must canonicalize `file` to repo-relative form before lookup.
Strategy: strip `project_root` prefix from an absolute `file`, or use `file`
as-is if already relative. Example:

```rust
let file_key = if file.is_absolute() {
    file.strip_prefix(project_root)
        .with_context(|| format!("{} is not under {}", file.display(), project_root.display()))?
        .to_path_buf()
} else {
    file.to_path_buf()
};
```

## Testing

### Post-implementation integration test (new file: `tests/integration_phase1d.rs`)

Requires network access (model download) — mark `#[ignore]` by default.

Scenario:
1. Create a tempdir with a git repo and a sample `.rs` file.
2. Call `service.index(&root, &root).await` — assert `files_indexed == 1`.
3. Call `service.search(&root, "fn main", 5).await` — assert non-empty results.
4. Call `service.overview(&root, Path::new("src/main.rs")).await` — assert
   non-empty string returned.
5. Modify the sample file, create a new commit, call `service.search` again —
   confirm implicit reindex triggers.

### Unit tests (no network, in `service.rs` `#[cfg(test)]`)

These can run offline:
- `ensure_index_fresh` on a fresh repo with no index file → verify `load_index`
  returns `None` and a full index is triggered (mock or use a tempdir with a
  tiny `.rs` file and a real git repo, but skip embed by ensuring the test repo
  has no `.rs` files that need embedding — or skip this as integration-only).

Most service-level correctness is better tested through the integration test.

## Success Criteria

- `cargo debrief search "ChunkType"` auto-indexes on first run, returns at
  least one result with `chunk.rs` in the path.
- `cargo debrief overview src/chunk.rs` returns the `Chunk`, `ChunkMetadata`,
  `ChunkKind`, `ChunkType`, `Visibility` overview chunks.
- `cargo debrief rebuild-index .` runs without error and prints files/chunks summary.
- Incremental reindex: modify one file, run search again — reindex touches only
  the changed file (verify by timing or log output).
- Index file is created at `.git/debrief/index.bin`.
- All existing tests still pass (`cargo test`).
